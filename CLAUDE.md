# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project shape

AutoScope is a minimal test harness for **web** (Playwright), **Android** (`uiautomator2` + `adb`), and **iOS** (WebDriverAgent via the `wda` client, Simulator or real device), plus a **desktop app** (Flet/Flutter) for recording and replaying web/Android/iOS tests, packaged for macOS/Windows/Linux. The iOS recorder/driver only works when the desktop app itself runs on macOS (Apple's tooling requirement, not ours) — the Web/Android Manual tabs are unaffected on other platforms.

Three usage modes:
1. **CLI**: `python -m autoscope.cli`
2. **Desktop recorder / runner**: `python run_desktop.py` (Flet UI)
3. **Generated standalone scripts**: `scripts/*.py` (written by the desktop recorder)

There is no pytest, no CI, and no lint/typecheck config in day-to-day use — tests are stdlib `unittest` plus a custom tag filter in `autoscope/core/runner.py`.

## Developer commands

```bash
# Install deps + Playwright browser
pip install -r requirements.txt
playwright install chromium   # or firefox / webkit

# Run all unittest-based tests discovered under current dir
python -m autoscope.cli run

# Run only web, Android, or iOS tests (tag filtering)
python -m autoscope.cli run --tag web
python -m autoscope.cli run --tag android
python -m autoscope.cli run --tag ios

# iOS: start WebDriverAgent against a booted Simulator first (one-time per boot)
./tools/start_ios_wda.sh

# Run a single test module/class by limiting the start dir + pattern
python -m autoscope.cli run --start-dir tests_web --pattern test_example_web.py

# Crawl a site starting from a URL (reports -> var/reports/crawl.html & crawl.json)
python -m autoscope.cli crawl --url https://example.com
python -m autoscope.cli crawl --url https://example.com --username alice --password secret
python -m autoscope.cli crawl --url https://example.com --max-depth 2 --max-pages 20

# Desktop app in dev mode
python run_desktop.py
# or
./run_desktop.sh

# Convenience wrapper for CLI
./run.sh [run|...]
```

Packaging (calls `flet build` under the hood; see `pyproject.toml [tool.flet]` for product/artifact names):

```bash
./packaging/build_macos.sh      # macOS: requires Xcode, CocoaPods -> build/macos/AutoScope.app + .dmg
.\packaging\build_windows.ps1    # Windows: requires VS "Desktop development with C++" -> build/windows/
./packaging/build_linux.sh      # Linux: requires GTK/build libs -> build/linux/
```

## Entry points

| Surface | File | Real entry |
|---|---|---|
| CLI | `autoscope/cli.py` | `main()` → `run_tests()` or `crawl()` |
| Desktop dev | `main.py`, `run_desktop.py`, `run_desktop.sh` | `autoscope.desktop.app:main` |
| Test base class | `autoscope/core/case.py` | `AutomateTestCase` |
| Web driver | `autoscope/drivers/web.py` | `WebDriver` (wraps Playwright) |
| Android driver | `autoscope/drivers/android.py` | `AndroidDriver` (wraps `uiautomator2` + `adb`) |
| iOS driver | `autoscope/drivers/ios.py` | `IOSDriver` (wraps `wda`, connects to an already-running WebDriverAgent) |
| Crawler | `autoscope/crawler.py` | `crawl()` / `save_crawl_report()` |

Package layout:
```
autoscope/
  config/       config loader
  drivers/      web, android, ios, adb wrappers
  core/         test base class + runner (discovery, tag filtering, reporting)
  reporting/    JSON + HTML reporters
  desktop/      Flet desktop app (recorder + auto-run script executor)
  cli.py        command-line entry point
tests_web/      web test packages
tests_android/  Android test packages
tests_ios/      iOS test packages (requires a running WebDriverAgent)
scripts/        scripts generated by the desktop recorder
```

## Configuration

- Default config file: `config.yaml`, loaded via `autoscope/config/loader.py`.
- Env overrides use these exact prefixes (not `TESTPILOT_`):
  - `AT_WEB_*` → maps to `config.yaml:web.*`
  - `AT_ANDROID_*` → maps to `config.yaml:android.*`
  - `AT_IOS_*` → maps to `config.yaml:ios.*`
  - `AT_RUNNER_*` → maps to `config.yaml:runner.*`
- Booleans are parsed (`true`/`false`) and digit-only values become ints; everything else is a string.
- In packaged desktop builds, `AUTOMATE_TESTER_HOME` points to the writable app-data dir and the packaged `config.yaml` is preferred over the bundled default (see `autoscope/desktop/paths.py`).

## Writing tests

- Inherit from `autoscope.AutomateTestCase` (in `autoscope/core/case.py`).
- Tag classes to control which drivers are provisioned:
  - `tags = ("web",)` → starts the web driver
  - `tags = ("android",)` → starts the Android driver
  - `tags = ("ios",)` → starts the iOS driver
  - `tags = ("web", "android", "ios")` → starts all three
  - Alternatively set class attrs `needs_web = True` / `needs_android = True` / `needs_ios = True`
- Inside a test:
  - `self.web` is a Playwright **`Page`** (not `WebDriver`). Use `self.web.goto(...)`, `self.web.page.fill(...)`, etc.
  - `self.android` is a `uiautomator2` **`Device`** (not `AndroidDriver`). Use `self.android(text="OK").click()`, etc.
  - `self.ios` is a `wda.Client` session (not `IOSDriver`). Use `self.ios.tap(x, y)`, `self.ios.swipe(...)`, `self.ios(name="Safari").click()`, etc.
  - `self.config` is the loaded `Config` dataclass.
- Screenshot helpers: `self.web_screenshot(name)`, `self.android_screenshot(name)`, `self.ios_screenshot(name)`.
- Failure screenshots are auto-captured to `var/reports/screenshots/` when `screenshot_on_failure` is enabled. Use distinct filenames across platform test files — they share one directory (see `tests_ios/test_example_ios.py`'s `ios_` prefix, added after it initially collided with `tests_android`'s identically-named screenshots).
- The runner (`autoscope/core/runner.py`) uses stdlib `unittest.TestLoader().discover()` plus `filter_tests_by_tag()` — tags are read off `test.__class__.tags`, not decorators or markers.

## Generated scripts & desktop app

- The desktop recorder (`autoscope/desktop/recorder/`) writes scripts to `scripts/`, for web, Android, and iOS (Web Manual, Android Manual, iOS Manual tabs).
- A generated script is a plain Python file with a `run()` function.
- Its platform is detected from a first-line comment, e.g. `# platform: web` or `# platform: android`.
- The desktop **Auto Run** tab executes these scripts in a subprocess via `autoscope/desktop/runner/script_runner.py` and writes `var/reports/auto_results.json` + `var/reports/auto_report.html`.
- Playwright browsers and `adb` are **not bundled** in packaged builds; the app checks for them at startup and can prompt to install Chromium.
- The desktop **Settings** tab lets the user pick custom scripts/reports folders (via a native OS folder dialog, `autoscope/desktop/folder_picker.py`), persisted to `desktop_settings.json` in the app data dir (`autoscope/desktop/settings.py`) and re-changeable at any time. `autoscope/desktop/paths.py`'s `get_scripts_dir()`/`get_reports_dir()` are the single source of truth every recorder/runner/view goes through, so they pick up the override automatically; "Reset to default" clears it.

## Diagnostics & debugging

One-off Playwright probes live at the repo root for quick debugging (not part of the test suite):

- `tools/diag_load.py <url>` — load a page and print status/title/URL.
- `tools/diag_login.py` — hardcoded target login flow probe (uses `CRAWL_USERNAME` / `CRAWL_PASSWORD`).
- `tools/diag_crawl_steps.py` — step-by-step login + link extraction probe.

## Gotchas

- No test framework beyond stdlib `unittest`; all discovery, tag filtering, and JSON/HTML reporting happens in `autoscope/core/runner.py`.
- Android tests require `adb` on PATH and a connected device. `device_serial: null` in config means "first available adb device".
- Web tests require the selected Playwright browser to be installed separately (`playwright install ...`).
- iOS tests require a WebDriverAgent instance already running and reachable at `config.yaml:ios.wda_url` (default `http://localhost:8100`) — `IOSDriver` connects to it, it does not build or launch WDA itself. For the Simulator, run `./tools/start_ios_wda.sh` once per boot (clones WebDriverAgent into `~/.autoscope/WebDriverAgent` on first run, then `xcodebuild test`s it — a multi-minute first build). Real devices need `tidevice` for code signing/USB tunneling instead. macOS + Xcode only; there is no Windows/Linux path for iOS automation.
- Crawl auto-login runs **only on the first page** and uses fallback selectors from `config.yaml` under `web.login_selectors`.
- Build scripts (`packaging/build_macos.sh`, `packaging/build_linux.sh`, `packaging/build_windows.ps1`) just call `flet build ...`; see `pyproject.toml [tool.flet]` for product/artifact names.
- Do not use `ft.FilePicker` (Flet 0.85.3): it renders as "Unknown control: FilePicker" on the macOS desktop client even on a freshly-downloaded dev client, no matter when/where it's added to `page.overlay` (a known, still-unresolved upstream bug — flet-dev/flet#6422, #6040). `autoscope/desktop/folder_picker.py` shells out to each OS's native folder dialog as a separate process instead (also avoids a second trap: tkinter's `askdirectory()` must run on the main thread on macOS, which Flet's own event loop already owns, so calling it from a worker thread crashes the whole process).
- Separately: a leftover `build/macos/AutoScope.app` (or `build/windows`, `build/linux`) from a previous `flet build` takes priority over the downloaded dev client in `ft.run()`/`run_desktop.py`. Since it's tailored to whatever controls were actually used *at that build time*, it can be missing ones added since — delete `build/<platform>/` when a control mysteriously shows as "Unknown control" in dev mode to rule this out first.

## Contribution conventions

- Prefer deletion over addition; keep code close to existing style; use the standard library over new dependencies when possible.
- Commit messages are short, present-tense, conventional-commit style (`feat:`, `fix:`, `docs:`).
- User-facing changes should update `CHANGELOG.md` under `[Unreleased]`.
