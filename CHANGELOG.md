# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

## [Unreleased]

### Changed
- **Consolidated `tests_web/`, `tests_android/`, `tests_ios/`, and `tests_core/` into one `tests/` package** (`tests/web/`, `tests/android/`, `tests/ios/`, `tests/core/`). Tag-based filtering (`--tag web/android/ios`) and default discovery (`unittest.TestLoader().discover()` from `.`) are both unaffected — tags come from `test.__class__.tags`, not directory layout, and discovery already recurses into any package with an `__init__.py`. Only `--start-dir tests_web`-style examples in the docs needed updating (now `--start-dir tests/web`).
- Deleted the leftover example scripts that had accumulated under `scripts/` from live-testing the recorders this session (`android_check_android.py`, `test_login_android.py`, `rerun_check_android.py`, `android_recording_test_android.py`, `test_login_web.py`) — these were throwaway artifacts of manual testing, not real examples worth keeping. `scripts/*.py` is now gitignored going forward, consistent with `var/` already being gitignored for reports: generated files from *using* the app were never meant to be committed.
- Reset the entire git history to a single commit. The repo had accumulated generated/throwaway files into tracked history before this cleanup's `.gitignore` existed; since old commits still carry those blobs regardless of what's ignored *now*, the only way to actually get rid of them was to start over rather than patch forward.
- **Renamed "Mobile" to "Android" throughout the codebase**, now that iOS support exists and "mobile" was ambiguous between the two. `MobileDriver`/`MobileConfig`/`MobileRecorder`/`MobileManualViewMixin` → `AndroidDriver`/`AndroidConfig`/`AndroidRecorder`/`AndroidManualViewMixin`; `autoscope/drivers/mobile.py` → `android.py`; `mobile_recorder.py` → `android_recorder.py`; `mobile_manual.py` → `android_manual.py`; `tests_mobile/` → `tests_android/` (later consolidated into `tests/android/`, see below). Tag `"mobile"` → `"android"` (`AutomateTestCase.needs_mobile`/`self.mobile`/`mobile_screenshot()` → `needs_android`/`self.android`/`android_screenshot()`). Config: `config.yaml`'s `mobile:` section → `android:`, env prefix `AT_MOBILE_*` → `AT_ANDROID_*`. Generated scripts: `# platform: mobile` → `# platform: android`, filename suffix `_mobile.py` → `_android.py` (existing example scripts under `scripts/` renamed and their content updated to match). Also dropped `MobileConfig.platform` (now `AndroidConfig`), a field that was never actually read anywhere and was doubly redundant once the config section itself is named `android`. This is a breaking change for anyone with an existing `config.yaml`, `AT_MOBILE_*` env vars, or hand-written tests tagged `"mobile"` — update them to the `android` equivalents.
- Desktop app now opens **maximized** by default instead of a fixed 1200×800 window.
- Replaced `ft.ResponsiveRow`/`col={...}` breakpoints with `ft.Row(wrap=True)` across every desktop view (Home's quick-action cards, and the toolbar rows in Web/Android/iOS Manual, Auto Run, and Reports). Root cause: `ResponsiveRow`'s Bootstrap-style breakpoints (xs/sm/md/lg/xl) are keyed to a reference width that doesn't line up with the actual space left after the nav rail, so at smaller window sizes it kept picking multi-column layouts that didn't leave enough room, squeezing content. `Row(wrap=True)` reflows based on the real measured width instead, so it degrades gracefully to fewer columns (down to one) at any window size instead of clipping/overlapping. Verified at maximized, 900×650, 720×540 (the clamped minimum), and 700×900 — clean reflow with no overlapping controls at any size.

### Added
- **Settings** desktop tab: lets the user choose custom folders for generated scripts and reports via a native OS folder dialog (`autoscope/desktop/folder_picker.py` — shells out to `osascript`/PowerShell/`zenity` per platform), with a "Reset to default" per folder. The choice persists to `desktop_settings.json` in the app data dir (`autoscope/desktop/settings.py`) and can be changed again at any time. `paths.py`'s `get_scripts_dir()`/`get_reports_dir()` are the single source of truth every recorder, `ScriptRunner`, and the Reports view already went through, so the override applies everywhere automatically with no other call site changes — this also fixed a latent inconsistency where the Reports view read `config.yaml`'s `runner.output_dir` while everything else read `paths.get_reports_dir()`, which could diverge in packaged builds. Note: changing a folder does not move existing files.
- **iOS support**: a new `IOSDriver` (`autoscope/drivers/ios.py`) connects to an already-running WebDriverAgent instance via the `wda` client — `self.ios` in `AutomateTestCase` (tag `"ios"` / `needs_ios = True`) is a `wda.Client` session supporting `.tap()`, `.swipe()`, `.screenshot()`, element queries, etc. New `IOSConfig` (`wda_url`, `bundle_id`, `screenshot_dir`, `video_dir`), `tests/ios/` example tests, and `tools/start_ios_wda.sh` to build/launch WebDriverAgent against a booted Simulator (clones into `~/.autoscope/WebDriverAgent` on first run). macOS/Xcode only (an Apple tooling requirement, not ours). Verified against a real Simulator: screenshot, tap (launched Safari, and separately Files, confirmed via `app_current()`), swipe, app launch/terminate by bundle ID, and text input all confirmed working end-to-end through the actual test harness.
- **iOS Manual** desktop recorder tab: a new `IOSRecorder` (mirrors `AndroidRecorder`) streams the Simulator/device screen (via `wda` screenshots) into the same tap/swipe/screenshot/video/Run-Test UI that Web Manual and Android Manual already have. Coordinates are mapped from the display container into WDA's *point* space (`window_size()`), not raw screenshot pixels, since that's what `tap()`/`swipe()` expect — the one real difference from the Android recorder. Video uses `xcrun simctl io <udid> recordVideo` (Simulator only), gracefully finalized via SIGINT. `ScriptBuilder` gained `build_ios_script()`; generated scripts call `session.tap()`/`session.swipe()`/`session.send_keys()` directly. Verified live end-to-end: connect & stream, tap (launched Files, confirmed via a clean single-session test after an earlier attempt was confounded by an incidental second WDA session — WDA only supports one active session at a time), stop & save, and Run Test all passed through the actual desktop app.
- Light/dark theme toggle in the desktop app navigation rail.
- Swipe gesture support in **Android Manual**: drag on the device preview to record and replay swipes, using the `AndroidRecorder.swipe()` / `ScriptBuilder` swipe support that already existed but had no UI to trigger it.
- Stream health feedback in **Android Manual**: after ~3s of failed screen captures (device unplugged, screen off, adb drop) the status line now says so instead of leaving a frozen, silently-dead preview; it clears automatically once frames resume.
- **Screenshot button** in both Web Manual and Android Manual: wires up `take_screenshot()`, which already existed on both recorders but had no UI control to trigger it during a manual session.
- **Video recording** for both Web Manual (Playwright's built-in `record_video_dir`) and Android Manual (`adb shell screenrecord`, gracefully finalized via SIGINT and pulled to `var/reports/videos/`), behind a new "Record video" checkbox in each view. New `WebConfig.video_dir` / `AndroidConfig.video_dir` config fields (default `var/reports/videos`).
- **Run Test button** in both Manual Recorder views: replays the just-saved script immediately via the same `ScriptRunner` instance the Auto Run tab uses, without leaving the recorder view.
- `tests/core/`: a new stdlib-`unittest` suite (56 tests, no browser/device required for most) covering `ScriptBuilder`, `config/loader.py`, HTML report escaping, `AndroidRecorder` coordinate math and video wiring, `adb` output parsing and screenrecord lifecycle, a real (headless) cross-thread Playwright regression test, and a headless smoke test that builds all five desktop views — the same class of check that would have caught the Flet API-drift bugs below automatically.

### Fixed
- Fixed the new Settings tab crashing the whole desktop app with an uncaught `NSInternalInconsistencyException` the first time "Change..." was clicked: `ft.FilePicker` itself turned out to be broken on this Flet version (see Gotchas in CLAUDE.md/AGENTS.md — "Unknown control: FilePicker" even on a fresh dev client, a known upstream bug), and the first fallback attempt, tkinter's `askdirectory()` run via `asyncio.to_thread`, crashed the process instead of showing a dialog because Tk requires the main thread on macOS, which Flet's event loop already owns. Settled on shelling out to each OS's native folder dialog as a fully separate process (`osascript`/PowerShell/`zenity`), which has no thread-affinity constraint. Verified live: dialog opens pre-navigated to the current folder, a chosen folder persists and updates the UI, and "Reset to default" correctly clears it back.
- Fixed `tests/ios/test_example_ios.py` using the exact same screenshot filenames (`device_connected.png`, `home_screen.png`) as `tests/android/test_example_android.py` — both write to the same shared `var/reports/screenshots/` directory, so running both suites would silently overwrite one platform's screenshot with the other's. Renamed to `ios_`-prefixed filenames.
- Fixed `config.yaml` never having gotten a `video_dir` line added for `web`/`android` when that feature was added earlier in this session, despite the file's convention of documenting every available field.
- Fixed **Web Manual**'s `Stop & Save` (and the new screenshot/video features) silently crashing: Playwright's sync API is thread-affine — every call on a browser/page must happen on the exact OS thread that created it. `start()` ran inside a throwaway thread that exited immediately after returning, so any later call from a different thread raised `cannot switch to a different thread (which happens to have exited)`. `WebRecorder` now runs its whole session (start/screenshot/stop) on one dedicated worker thread via a small command queue, so it stays safely callable from any thread.
- Fixed a layout bug where the new "Run Test" button in Android Manual was squeezed into a 1/12-width column, wrapping its text vertically into an unreadable circle.
- Fixed HTML/crawl reports embedding unescaped test names, messages, and tracebacks (`autoscope/reporting/html.py`) — a crawled page with `<script>`/`<img onerror=...>` in its title or an exception message could inject markup into a report opened directly in a browser. Now escaped with `html.escape()`.
- Fixed `AT_WEB_*` / `AT_ANDROID_*` / `AT_RUNNER_*` environment variable overrides silently doing nothing unless the corresponding key already existed in `config.yaml` — `_env_override` only iterated keys already present in the loaded YAML dict instead of the full dataclass schema. Env overrides now apply regardless of what's in config.yaml, matching the documented behavior.
- Fixed the Android recorder never showing the streamed device screen and dropping every tap/text input: `page.run_task()` requires an `async` coroutine in Flet 0.85 and was being called with plain sync functions (raised `TypeError`, silently swallowed in the streaming loop); tap coordinates were read from a nonexistent `TapEvent.local_x/local_y` (now `local_position.x/y`); and the live frame was assigned to a nonexistent `Image.src_base64` (now `Image.src`, which accepts a base64 data URI directly).
- Fixed a startup/runtime crash from `ft.Colors.SUCCESS` / `ft.Colors.WARNING`, which don't exist in this Flet version (Material3 has no built-in success/warning role) — replaced with `ft.Colors.GREEN` / `ft.Colors.AMBER`. This fired unconditionally whenever `adb` wasn't detected, before the main window ever appeared.
- Fixed generated recorder scripts being invalid Python (`SyntaxError`: empty `try:` block) when a manual recording session captured zero actions; `ScriptBuilder` now emits `pass` as a fallback body. Repaired three previously-committed scripts under `scripts/` that were broken this way.
- Fixed `config/loader.py` dataclass defaults still pointing at the old `reports/` path after the project-wide move to `var/reports/`; a missing config.yaml would silently resurrect the old layout.
- Fixed desktop app home page appearing empty on startup; navigation now refreshes the whole page once after building each view.
- Fixed **Android Manual** and **Auto Run** tabs failing to open due to Flet 0.85 API changes (`ft.Image` now requires `src`, `ft.alignment.center` replaced with `ft.Alignment.CENTER`, snackbar uses `page.overlay`).
- Fixed **Auto Run** script list throwing "Control must be added to the page first" during initial render.
- Widened the navigation rail so all labels, including "Android Manual", stay centered on a single line.
- Made cross-platform build scripts (`build_macos.sh`, `build_linux.sh`, `build_windows.ps1`) non-interactive and safe for CI by adding `--yes` and `--no-rich-output` flags.
- Bypassed rich Windows console crash on CI runners by piping `flet build` output through a log file and setting UTF-8 console/env encoding.

## [1.0.0] - 2026-07-04

### Added
- Initial release of AutoScope: a minimal test harness for web (Playwright) and Android mobile (uiautomator2 + adb).
- Desktop recorder / runner built with Flet for manual web and mobile test recording.
- CLI for running stdlib `unittest`-based tests with tag filtering (`web`, `mobile`).
- No-script crawl mode with optional auto-login and JSON/HTML reports.
- Screenshot-on-failure helpers for both web and mobile tests.
- Config loading from `config.yaml` with env overrides using `AT_WEB_*`, `AT_MOBILE_*`, and `AT_RUNNER_*` prefixes.
- Packaging scripts for macOS (`.app` + `.dmg`), Windows (`.exe`), and Linux.
